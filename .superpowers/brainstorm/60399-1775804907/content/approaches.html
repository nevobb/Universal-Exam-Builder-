3 Architectural Approaches for the STEM Exam Viewer

Choose the structure that feels right for this codebase

A

Single File — ExamViewer.jsx

One self-contained file with the Pyodide hook, QuestionCard, DiagramViewer, and JSON input — all co-located. Imported into Architecture.jsx as a new tab. Matches the existing "one big file" culture perfectly.

Pros

  • Consistent with Architecture.jsx style
  • Zero extra files to wire together
  • Easy to hand off to the AI for edits

Cons

  • File will be 400–600 lines
  • All concerns in one place
B ⭐ Recommended

Modular — app/src/viewer/ folder

4 small, focused files: ExamViewer.jsx (page), usePyodide.js (hook), QuestionCard.jsx, DiagramViewer.jsx. Architecture.jsx imports only the top-level ExamViewer. Clean separation, each file has one job.

Pros

  • Each unit is small and understandable
  • Easy to test DiagramViewer in isolation
  • Best for future growth

Cons

  • More files than the existing pattern
  • Requires a new directory
C

Inline Expansion — Add directly to Architecture.jsx

All viewer logic added as new sections at the bottom of the existing Architecture.jsx monolith. Zero new files, maximum integration with existing state/theme/icons.

Pros

  • Shares icons, tokens, theme hooks for free
  • No import wiring needed

Cons

  • Architecture.jsx already 1,218 lines — grows to ~1,800+
  • Harder to reason about
  • You explicitly asked for "own component file"

My Recommendation

Option B — Modular viewer folder. The 4-file structure gives each component a clear single purpose, keeps files readable, and makes Pyodide's async lifecycle completely isolated from the rest of the app. Architecture.jsx stays clean.
